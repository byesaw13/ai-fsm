## BREAKING — Fix before next refactor

### B1: Sub-status names broken in pipeline stage derivation

**Location:** `apps/web/lib/pipeline/stages.ts:88-89` vs `packages/domain/src/sub-statuses.ts:3,10`

The `derivePipelineStage()` function checks for sub-statuses `waiting_customer` and `waiting_weather` to derive the `waiting` stage:

```ts
// lib/pipeline/stages.ts — WRONG names
facts.subStatus === "waiting_customer" ||   // ← does not exist
facts.subStatus === "waiting_weather"        // ← does not exist
```

The actual sub-status values are:

```ts
// packages/domain/src/sub-statuses.ts — CORRECT names
"customer_hold"   // not waiting_customer
"weather_hold"    // not waiting_weather
```

**Impact:** A job with `customer_hold` or `weather_hold` sub-status never derives the `waiting` pipeline stage. It falls through to `in_progress`, `scheduled`, or another stage instead. The `waiting` stage is only reachable via `waiting_parts`. The `Waiting` column on the pipeline board is effectively dead for two of its three intended triggers.

**Fix:** Change `pipeline/stages.ts` to use `customer_hold` and `weather_hold`.

## MAJOR — Resolve before adding UI complexity

### M1: Two parallel stage systems for the same concept

**Locations:**
- `packages/domain/src/stages.ts` — `CustomerStage` (5 stages)
- `apps/web/lib/pipeline/stages.ts` — `PipelineStage` (10 stages)

Both describe job lifecycle from a presentation perspective. Neither is stored in the DB.

| System | Stages | Used by |
|---|---|---|
| `CustomerStage` | intake, estimate, accepted, scheduled, completed | Job board (`/app/jobs`), portal, `deriveCustomerStage()` |
| `PipelineStage` | new_lead, estimate_needed, estimate_sent, approved_ready, scheduled, in_progress, waiting, completed, invoiced, archived | Pipeline page (`/app/pipeline`), job detail command panel, `derivePipelineStage()` |

**Impact:** Two derivation functions, two sets of stage labels, two cognitive models for the same lifecycle. `JobBoard.tsx` accepts `groupBy: "status" | "pipeline_stage"` — it's already trying to bridge them. Adding features to one requires auditing the other.

**Recommendation:** Consolidate into a single `OperationalStage` in the domain package. `CustomerStage` (portal-facing, 5 stages) should remain separate as a simplification of `OperationalStage`, not a peer. This means one derivation function and one label set internally, with a slim projection for the portal.

## MEDIUM — Clean up during consolidation phase

### C1: `vault_item` and `membership` event types are dead code in PropertyTimeline

**Location:** `apps/web/app/app/properties/[id]/PropertyTimeline.tsx:4`

The TypeScript type defines 9 event types:

```ts
type TimelineEventType = "visit" | "estimate" | "invoice" | "vault_item" | "membership" | "equipment" | "photo" | "issue" | "note";
```

But `property_timeline_v` (the SQL view, `db/migrations/057_property_timeline_view.sql`) only emits 7:
`visit`, `estimate`, `invoice`, **`equipment`**, `photo`, `issue`, `note`

- `vault_item` is **never emitted** by the view — dead type
- `membership` is **never emitted** by the view — dead type
- `equipment` is what the view uses for vault items (from `property_vault_items` table)

**Secondary issue:** The domain model calls these "vault items" but the timeline view calls them `equipment`. This is a naming inconsistency between the view and the domain vocabulary. The UI chips say "Equipment" but the canonical term is "Vault Item" or "Home Vault."

**Fix (two steps):**
1. Remove `vault_item` and `membership` from the `TimelineEventType` union
2. Rename `equipment` → `vault_item` in the view SQL and all consuming code, or update the domain model to acknowledge `equipment` as the canonical timeline event name

### C2: Legacy scheduling fields still being written

**Locations where legacy fields are still written:**
- `apps/web/app/api/v1/jobs/route.ts:59-86` — POST accepts `scheduled_start`, `scheduled_end` and inserts them

**Locations where legacy fields are still read:**
- `apps/web/app/api/portal/clients/[token]/route.ts:56` — reads `j.scheduled_start, j.scheduled_end`
- `apps/web/app/portal/[clientToken]/page.tsx:119` — reads `j.scheduled_start, j.scheduled_end`

The schema comment at `db/migrations/001_core_schema.sql:80` explicitly says:
```sql
-- LEGACY: visits are the source of truth for scheduling.
-- Do not read these fields for display; derive scheduling from visits.scheduled_start instead.
```

**Impact:** The portal shows scheduling info from `jobs.scheduled_start` which could be stale (set at job creation, not updated when visits change). The actual scheduled time lives on `visits.scheduled_start`.

**Fix:**
1. Remove `scheduled_start/scheduled_end` from the job POST body schema
2. Rewrite the two portal queries to derive schedule from visits (`LEFT JOIN visits`)
3. Future migration: drop the columns

### C3: Two automation tables, one misleading name

**Tables:**
- `automations` (from `001_core_schema.sql`) — scheduler table: `type` (visit_reminder | invoice_followup), `enabled`, `config`, `next_run_at`, `last_run_at`
- `automation_rules` (from `061_automation_rules.sql`) — governance config: `cooldown_hours`, `max_per_day`, `working_hours_start`, `working_hours_end`, `suppress_on_open_invoice`

These are fundamentally different things with confusingly similar names:
- `automations` = the actual automation jobs that run on a schedule
- `automation_rules` = rate-limiting and throttling configuration (one row per account)

**Impact:** `automation_rules` is not a rules engine. It's a settings table. Calling it `automation_rules` implies it contains triggering conditions, which it doesn't.

**Recommendation:** Rename `automation_rules` → `automation_settings` (or `notification_settings`) to clarify its purpose. Update API documentation and in-app label from "Automation Rules" to "Notification Settings" where this config is displayed.

### C5: `My Day` and `Field` pages overlap without clear differentiation

**Locations:**
- `apps/web/app/app/my-day/` — 163 lines, shows all visits (including completed), today/upcoming/overdue/past, richer metadata, role-aware
- `apps/web/app/app/field/` — 136 lines, shows only active visits (not cancelled/completed), today/upcoming split, shows active time tracking

Both appear in the nav sidebar under Operations for tech users. Both answer "what visits do I work on?"

**Distinction that exists but isn't communicated:**
- My Day = planning/overview view (before you leave the office)
- Field = in-the-field execution view (while on site)

**Impact:** Tech users have two very similar pages with no clear mental model for when to use each. Field page even has a "My Day" button in its header, suggesting the relationship is known but not surfaced to users.

**Recommendation:** Make the distinction explicit in UI copy. Consider: My Day = "Today's Schedule" (full day overview), Field = "On Site" (active execution, shown when a visit is in progress). Or merge them into one adaptive page.

## What is NOT an issue

These were suspected overlaps but are intentional and correct:

- **`job_type` vs `job_category`** — These are genuinely different fields. `job_type` is the work category (what kind of trade). `job_category` is the strategic acceptance category (why you took the job). Both fields should exist. The naming is slightly confusing but not wrong.

- **`maintenance_plans` table + `membership` product language** — Table name is preserved for migration compatibility; product language in UI correctly says "Membership." This is the right approach. No conflict.

- **`assets` table (Homebox) vs vault items** — These are genuinely different things. Homebox asset links track Dovetails' own tools/equipment used on jobs. Vault items track the homeowner's appliances and systems. The asset link (`entity_type: "job" | "visit"`) confirms it links Dovetails assets to jobs, not homeowner property facts.

- **`intake/new` route vs `booking-requests` route** — These are different surfaces. `/intake/new` is an admin form for manually entering a new request. `/booking-requests` is the queue of existing requests. The naming inconsistency (one says intake, one says booking-request) is minor and tolerable.
